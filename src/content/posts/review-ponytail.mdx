## The pitch

Ponytail is a [Claude Code](/posts/review-claude-code) skill that loads as a SessionStart hook and
tells the agent to behave like the laziest senior developer in the room, on the theory that the best
code is the code you never write. It is not a prompt you have to remember to invoke. The hook fires
every session, the instruction sits in context for every response, and a statusline badge shows
which intensity is live. There are three: lite, full, and ultra. Full is the default and the one I
ran.

The mechanism is a ladder, and the agent has to stop at the first rung that holds:

1. Does this need to exist at all?
2. Can the standard library do it?
3. Does a native platform feature cover it?
4. Can an already-installed dependency handle it?
5. Can it be one line?
6. Only then, the minimum code that works.

Around the ladder sit a handful of rules with teeth. No abstraction with a single implementation. No
scaffolding for a "later" that can scaffold for itself. Deletion over addition. When it does cut a
corner on purpose, it is told to leave a `// ponytail:` code comment naming the shortcut and when to
revisit it, so the next reader sees an intentional choice rather than a lazy miss. As an idea, this
is good. I want my agent to think this way. The problem is not the philosophy, it is everything the
plugin does to make sure you never forget it is installed.

## What it looks like in a real run

Give the engine its due first, because it works. I pointed it at a slow build with a ticket that
invited a full toolchain swap, and the laziest path turned out to be the best one: a transpiler swap
that cut the cold build 63% without moving the orchestrator or the bundler. The
[full case study](/posts/a-faster-build-without-the-rewrite) is its own post. What the ladder
bought, concretely:

- It rejected both tools my ticket named and changed one layer instead of two foundations.
- It wrote the same loader rule into all 28 configs and held off on the shared factory until the
  migration was proven, then built it once. Speculative abstraction is exactly what the top rung
  kills.
- Told to clean up runtime Babel dependencies, it scanned 39 package manifests, found none in the
  wrong place, and changed a single doc. The lazy answer to "do this cleanup" was "there is nothing
  to clean up, here is the proof."
- It deleted a source-map loader rather than configure it, and pinned a freshly released loader
  version rather than float to a same-day release it had not tested.

None of those are dramatic, and that is the point. The bias shows up as a hundred small refusals to
do more than the task needs, and they add up to a diff you can actually review. If the story ended
here, this would be a four-star review.

## The tax

The story does not end there. Every session begins with Ponytail dumping its entire persona into the
transcript: a `PONYTAIL MODE ACTIVE` banner followed by the full specification of the ladder, the
rules, the intensity table, the boundaries, all of it, as a wall of text you did not ask to read and
cannot dismiss short of uninstalling. That same persona then rides along in context for every
response you get for the rest of the session, quietly spending tokens to keep reminding a model of a
preference a single line of instruction already covers.

Then there is the part that tipped me from annoyed to done. The plugin ships a statusline hook whose
setup message instructs your agent, verbatim, to "proactively offer to set this up for the user on
first interaction." Read that again. A tool I installed to make my agent lazier is using my agent as
an unpaid salesman for its own statusline badge. I did not ask to be marketed to by software running
inside my own terminal, least of all through the assistant I am paying to do work. It is a small
thing and it is exactly the kind of small thing a plugin about restraint should have been incapable
of shipping.

Removal is the last insult. A tool this eager to install itself everywhere leaves itself everywhere:
a plugin cache, a cloned marketplace, a data directory, a loose `.ponytail-active` file in your
Claude home, and three separate references across the plugin registries and your settings. Turning
it off is not a toggle, it is a scavenger hunt. For a plugin whose entire pitch is deleting things
you do not need, it is remarkably hard to delete.

## Where the credit belongs

A fair review has to draw this line, and it is the same line that sinks the score. What kept the
build job small was the agent's own analysis in a planning session. A capable agent with a decent
`CLAUDE.md` already leans this way. Most of what Ponytail encodes (no unrequested abstractions,
delete dead code, confirm before adding a dependency) is what careful engineers already write into
their standing instructions, quietly, in a few lines, without a hook that narrates itself.
Ponytail's contribution is making that default loud. The trouble is that "loud" is the whole
product, and loud is precisely what you do not want from a tool whose thesis is doing less.

## Where it bites

Beyond the noise, it governs what the agent builds, not how honestly it checks the result. More than
once the same lazy reflex turned into an unverified claim: a secrets file reported as "missing" that
one `ls` proved was sitting right there, an experiment called clean before its harness was even
valid. A human caught each one. The "leave a runnable check behind" rule is about the code it
writes, not the claims it makes, so it does nothing to stop an agent declaring something done
without looking. The `ponytail:` comments have the same problem:
[this PR](https://github.com/meese-os/meeseOS/pull/202) shipped without a single one, while a
separate run left exactly the comment the skill asks for. A convention the agent honors only when it
remembers is not one you can count on.

## The rest of the kit

The one part worth keeping is the commands. `/ponytail-review` is a code review that hunts only for
over-engineering: reinvented standard library, dead flexibility, dependencies you did not need.
`/ponytail-audit` runs the same lens across a whole repo and ranks what to delete. `/ponytail-debt`
harvests every `ponytail:` comment into a ledger. Those are useful, on-demand, and mercifully quiet,
which is everything the always-on mode is not.

Roughly thirty thousand GitHub stars usually means a good idea, and there is a good idea in here.
There is also a SessionStart lecture, a resident context tax, and a hook that turns your assistant
into a billboard for the plugin. Lift the ladder into your `CLAUDE.md`, keep the audit command
bookmarked if you like it, and skip the rest.
