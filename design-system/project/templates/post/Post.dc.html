Getting an Expo SDK 56 app running on a real device

Modernizing an old app to the newest SDK turns up a few gotchas the docs never connect for you. Here is the path that actually worked, in order, with the errors you will see along the way.

react-native expo mobile-dev

The upgrade itself is one command. Everything that breaks afterward is a version-mismatch problem wearing a different mask each time — the JS bundle, the dev client, and the native runtime all have to agree, and the error you get rarely names the one that disagreed.

Start from a clean dev client

Before touching the SDK version, rebuild the development client so you are debugging one change at a time. A stale dev client is the single most common reason a “successful” upgrade still red-screens on launch.

npx expo install expo@^56.0.0
npx expo prebuild --clean
eas build --profile development --platform android
If you install SDK 56 but launch yesterday’s dev client, the runtime mismatch surfaces as an opaque bundling error. Rebuild the client first, then load JS into it.

Read the error, not the stack

The runtime points at the JS bundle, but the real problem is usually the native side that loaded it. When you see a mismatch error, check the installed expo version against the dev client’s embedded runtime before you touch anything in metro.config.js.

The fastest path through an SDK upgrade is to stop trusting the first error and verify the three versions agree by hand.

Once the three agree, the rest of the upgrade is ordinary: bump the libraries Expo flags, run the app on device, and fix the handful of API renames the changelog lists explicitly.
